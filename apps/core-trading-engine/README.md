# Core Trading Engine

This app is the in-memory matching and settlement engine for the centralized exchange. It receives typed engine events, validates them through the OMS layer, mutates the engine state, settles balances or perp positions, returns a backend-safe response, and snapshots the state after successful mutating events.

The engine currently supports:

- Spot order processing.
- Perpetual order processing.
- Market, limit, GTC, IOC, and FOK behavior.
- Post-only checks.
- Self-trade prevention.
- Price-time priority matching.
- User balances with locked funds.
- Perp margin locking and position updates.
- Order, depth, market, balance, and open-order queries.
- Snapshot save and restore.

## Runtime Entry Point

The process starts in `src/index.ts`.

```ts
const engine = new Engine();
const nats = await NatsManager.getInstance();
await nats.subscribe("engine.>", engine.process);
```

Incoming NATS subjects under `engine.>` are passed directly to `engine.process(subject, payload)`. The `subject` decides which engine operation runs, and the `payload` is cast to the matching request type from `@workspace/types`.

There is also a commented Redis-stream path in `src/index.ts`. That path is intended to consume market events from Redis, process them through the same `engine.process(...)` method, and publish the result back to a backend stream. The important design point is that both NATS and Redis route through the same core engine API.

## Main Files

- `src/index.ts`: starts the engine and subscribes to incoming transport events.
- `src/engines/core-engine.ts`: orchestrates every request and owns the shared `EngineState`.
- `src/engines/oms-engine.ts`: validates incoming orders, market operations, user operations, and risk checks.
- `src/engines/matching-engine.ts`: thin router from market id to the correct single-market orderbook.
- `src/engines/single-orderbook.ts`: actual matching engine for one market.
- `src/engines/balance-engine.ts`: user balance creation, deposits, locking, unlocking, and spot settlement.
- `src/engines/position-engine.ts`: thin router from market id to the correct single-market position store.
- `src/engines/single-market-positions.ts`: perp position indexes, settlement, margin release, realized PnL, and liquidation-price lookup for one market.
- `src/engines/market-engine.ts`: market setup, market CRUD, and supported base/quote assets.
- `src/utils/parse-incoming.ts`: converts string amounts into `bigint` for calculation and converts responses back into strings for JSON.

## Engine State

`EngineState` is the shared in-memory state object passed into all sub-engines.

```ts
balances: Map<UserId, Map<Asset, { total: bigint; locked: bigint }>>
orderbooks: Map<MarketId, SingleMarketOrderBook>
positions: Map<MarketId, SingleMarketPositions>
markets: Map<MarketId, Market>
orderMap: Map<OrderId, MarketId>
orders: Map<OrderId, InMarketOrderType>
```

Each map has a specific responsibility:

- `balances` stores total and locked balances per user and asset.
- `orderbooks` stores one `SingleMarketOrderBook` per market.
- `positions` stores one indexed `SingleMarketPositions` instance per market.
- `markets` stores market metadata like base asset, quote asset, tick size, lot size, and max leverage.
- `orderMap` maps active resting order ids to their market id, so cancel/get operations can find the book quickly.
- `orders` stores only active resting orders. Matching returns transient maker and taker references for immediate settlement without retaining filled or cancelled orders globally.

## Event Flow

Every event enters through:

```ts
Engine.process(subject, data)
```

`process` switches on `EVENT_TO_ENGINE_SUBJECT` and calls the matching private method.

Read-only events return directly:

- `HEALTH_CHECK`
- `ORDER_OPEN_ORDERS`
- `ORDER_GET`
- `BALANCE_GET`
- `DEPTH_GET`
- `MARKET_GET_ALL`
- `MARKET_GET`

Mutating events are saved to snapshot after a successful result:

- `ORDER_CREATE`
- `ORDER_CANCEL`
- `ON_RAMP`
- `USER_ADD`
- `MARKET_ADD`
- `MARKET_UPDATE`
- `MARKET_DELETE`
- `MARKET_ADD_ASSET`

Every response includes:

- `success`
- `eventId`
- `timestamp`
- `message`
- optional `userId`
- optional `code` for rejected requests
- optional `data`

`eventId` is generated by `getUpdatedEventId()` and increases once per returned engine response.

## Order Creation Flow

Order creation is the most important path.

```txt
NATS/Redis/backend payload
  -> Engine.process(ORDER_CREATE, payload)
  -> Engine.createOrder(payload)
  -> OMSEngine.createOrderChecks(payload)
  -> BalanceEngine.lockBalance(parsedOrder)
  -> MatchingEngine.createOrder(parsedOrder)
  -> SingleMarketOrderBook.addOrder(order)
  -> fills are settled
  -> unused locked balance is released
  -> order response is normalized
  -> snapshot is saved
```

### 1. Payload Normalization

Incoming order payloads use strings for price, quantity, and margin because JSON cannot represent `bigint`.

`normalizeOrderIncoming(...)` converts:

- `quantity: string` to `quantity: bigint`
- `entryPrice: string` to `entryPrice: bigint`
- `margin: string` to `margin: bigint` for perp orders

The engine does all money and quantity math using `bigint`.

### 2. OMS Checks

`OMSEngine.createOrderChecks(...)` performs the validation pipeline:

1. Validate market exists.
2. Validate order side, type, quantity, and price.
3. Validate time-in-force rules.
4. Validate market constraints.
5. Validate perp position rules.
6. Validate orderbook rules.
7. Validate risk and balance/margin availability.

Basic checks include:

- Side must be `BUY` or `SELL`.
- Type must be `LIMIT` or `MARKET`.
- Quantity must be positive.
- Limit orders require `entryPrice`.
- Market orders currently also require an `entryPrice` for risk checks.
- Price must be positive.
- Perp orders require a valid `position` and positive `leverage`.
- Spot orders cannot include a perp position direction.

Time-in-force checks include:

- Market orders cannot use GTC.

Market constraints include:

- Perp leverage cannot exceed market `maxLeverage`.
- Quantity must be at least `minQty`.
- Quantity must follow `lotSize`.
- Price must follow `tickSize`.
- Notional must be at least `minNotional`.

Orderbook checks include:

- The market must have an orderbook.
- User open orders in a market are capped.
- Post-only limit orders must not immediately cross the book.
- Self-trade prevention rules are checked against existing opposite-side user orders.

Risk checks include:

- Market orders require enough opposite-side liquidity.
- Spot buy requires available quote asset.
- Spot sell requires available base asset.
- Perp order requires available quote collateral for initial margin.

### 3. Balance Locking

Before matching, the engine locks the maximum amount that could be needed.

For spot:

- Buy locks `entryPrice * quantity` of the quote asset.
- Sell locks `quantity` of the base asset.

For perps:

- The engine locks required initial margin in the market quote asset:

```txt
requiredMargin = entryPrice * quantity / leverage
```

If matching fails after locking, `createOrder` calls `releaseBalance(...)` to unlock the amount and avoid stranded locked funds.

### 4. Matching

`MatchingEngine.createOrder(...)` creates an internal order with:

- generated `orderId`
- `filled = 0`
- `remainingQty = quantity`
- `averagePrice = 0`
- `status = OPEN`
- empty `fills`
- empty `depths`

Then it routes to:

```ts
SingleMarketOrderBook.addOrder(order)
```

The single-market book stores:

- `bids`: price level to FIFO order list.
- `asks`: price level to FIFO order list.
- `bidTree`: sorted bid prices.
- `askTree`: sorted ask prices.
- `orderMap`: active order id to order node for this market.
- `userOrders`: user id to active order ids for this market.

The maps hold the orders. The red-black trees make best bid and best ask lookup efficient.

### 5. FOK, Post-Only, and Crossing

Before matching:

- FOK calculates the executable quantity across currently crossable opposite levels. If the full remaining quantity cannot execute, the order is cancelled without trading.
- Post-only checks whether the order would cross the best opposite price. If it would, the order is rejected.

Crossing rules:

- Buy limit crosses when `buyPrice >= bestAsk`.
- Sell limit crosses when `sellPrice <= bestBid`.
- Market order crosses any available opposite price.

### 6. Price-Time Priority

Matching always starts at the best opposite price:

- A buy order matches the lowest ask.
- A sell order matches the highest bid.

Inside one price level, orders are matched FIFO through `OrderList`. This gives price-time priority:

1. Better price first.
2. Earlier order first at the same price.

For each match:

- `tradeQty = min(takerRemaining, makerRemaining)`
- maker and taker `filled` increase
- average prices are updated
- a fill is emitted
- price level `totalQty` decreases
- fully filled makers are removed from the book

Fill shape:

```ts
{
  tradeId,
  makerOrderId,
  takerOrderId,
  makerUserId,
  takerUserId,
  side,
  marketId,
  qty,
  price,
  timestamp,
  status: "TRADE"
}
```

### 7. Self-Trade Prevention

If maker and taker have the same `userId`, `handleSTP(...)` applies the taker STP mode:

- `CANCEL_TAKER`: taker is cancelled and matching stops.
- `CANCEL_MAKER`: maker is cancelled and removed, then taker continues.
- `CANCEL_BOTH`: both orders are cancelled and matching stops.

### 8. Resting Remaining Quantity

After matching:

- Fully filled orders become `FILLED`.
- Market, IOC, and FOK orders with remaining quantity do not rest. They become `PARTIAL_FILLED` if any fill happened, otherwise `CANCELLED`.
- Limit GTC orders with remaining quantity rest on the book. They become `OPEN` if no fill happened or `PARTIAL_FILLED` if some quantity filled.

Resting orders are added to:

- the side map (`bids` or `asks`)
- the side tree (`bidTree` or `askTree`)
- market-local `orderMap`
- global `orderMap`
- market-local `userOrders`

The global `orders` map stores the same active resting orders. Terminal orders
are returned to the caller but are not retained globally.

### 9. Settlement

After matching, `Engine.createOrder(...)` loops over the returned fills.

For spot:

```ts
balanceEngine.applyFill(fill, makerOrder, takerOrder)
```

For perps:

```ts
positionEngine.applyFill(fill, makerOrder, takerOrder)
```

Then:

```ts
balanceEngine.releaseUnusedBalance(result)
```

This releases unneeded locked funds for IOC, FOK, market orders, fully filled orders, partially filled orders, or price improvement.

## Spot Settlement

Spot settlement uses transient maker and taker references returned by matching.

For a buy fill:

- Debit quote from locked balance: `qty * price`.
- Credit base total: `qty`.

For a sell fill:

- Debit base from locked balance: `qty`.
- Credit quote total: `qty * price`.

Example:

```txt
Seller rests: SELL 2 BTC @ 100 USD
Buyer crosses: BUY 2 BTC @ 100 USD

Seller:
  BTC total -2
  BTC locked -2
  USD total +200

Buyer:
  USD total -200
  USD locked -200
  BTC total +2
```

Because the engine locks before matching, settlement debits locked balances instead of directly debiting available balances.

## Perp Settlement and Positions

Perp fills go to the `Position` routing wrapper, which delegates to the
market's `SingleMarketPositions` instance.

The position engine loads:

- market data
- maker order
- taker order
- indexed single-market position store

For each fill it applies the fill to both users.

If the user has no position:

- A new position is created.
- Position side is taken from the order position if present, otherwise inferred from order side.
- Average price and entry price are set to the fill price.
- Margin is `qty * price / leverage`.
- Bankruptcy price marks full isolated-margin exhaustion.
- Liquidation price triggers 10% of the margin-loss distance before bankruptcy.

If the user already has the same position direction:

- Quantity increases.
- Average price is recalculated by weighted average.
- Margin increases by fill margin.
- Leverage is updated from the order.
- Liquidation and bankruptcy prices are recalculated and reindexed.

If the fill is opposite the current position:

- The engine calculates closing quantity.
- Margin is released proportionally.
- Realized PnL is calculated:

```txt
LONG PnL  = (exitPrice - entryPrice) * qty
SHORT PnL = (entryPrice - exitPrice) * qty
```

Then:

- If the fill is smaller than the current position, the position is reduced.
- If the fill equals the current position, the position is closed and removed.
- If the fill is larger, the old position closes and a new opposite position is opened with the extra quantity.

## Cancel Flow

```txt
Engine.process(ORDER_CANCEL, payload)
  -> OMSEngine.cancelOrderChecks(payload)
  -> MatchingEngine.cancelOrder(payload)
  -> SingleMarketOrderBook.cancelOrder(payload)
  -> BalanceEngine.releaseOrderMargin(order)
  -> normalize response
  -> save snapshot
```

Cancel validation checks the order id exists and belongs to a live order node.

The orderbook cancel path:

1. Finds the market through global `orderMap`.
2. Finds the order node inside the market orderbook.
3. Verifies the `userId`.
4. Removes the order from the price level FIFO list.
5. Removes the price level from the tree if empty.
6. Removes the order from order maps and user open orders.
7. Marks status as `CANCELLED`.

After cancellation, locked balance is released:

- Spot buy releases quote for `entryPrice * remainingQty`.
- Spot sell releases base for `remainingQty`.
- Perp releases margin for the remaining quantity.

## Balance Flow

### Add User

`USER_ADD` creates a balance map for the user and initializes every asset currently known from the markets.

### On-Ramp

`ON_RAMP`:

1. Normalizes `amount` from string to `bigint`.
2. Validates the asset exists in at least one configured market.
3. Adds the amount to the user's total balance.
4. Leaves locked balance unchanged.

### Get Balances

`BALANCE_GET` returns all balances for a user with `total` and `locked` converted back to strings.

## Market Flow

`MarketEngine.initializeMarkets()` creates default markets by combining:

```ts
baseAsset = ["BTC", "ETH", "SOL"]
quoteAsset = ["USD", "INR", "PERP"]
```

That creates markets like:

- `BTC_USD`
- `BTC_INR`
- `BTC_PERP`
- `ETH_USD`
- `SOL_PERP`

For every market, the engine creates:

- market metadata
- an empty positions map
- a `SingleMarketOrderBook`

Market operations:

- `MARKET_GET_ALL`: returns all markets.
- `MARKET_GET`: validates and returns one market.
- `MARKET_ADD`: adds market metadata, positions map, and orderbook.
- `MARKET_UPDATE`: updates mutable market constraints, but rejects base/quote changes.
- `MARKET_DELETE`: rejects deletion when the market has open orders, positions, or user balances.
- `MARKET_ADD_ASSET`: adds a new base or quote asset to the supported asset list.

## Depth Flow

`DEPTH_GET` validates the market and then calls `orderbook.getDepth()`.

Depth is built from the trees:

- Bids iterate from highest price downward.
- Asks iterate from lowest price upward.
- Each level returns price and aggregate quantity.

Default depth limit is 20 levels.

## Order Queries

### Get Order

`ORDER_GET` validates the order id and returns the order. The matching engine first checks the global `orders` map so filled/cancelled history can still be returned when available.

### Get Open Orders

`ORDER_OPEN_ORDERS` validates the market and returns the user's active order ids from the market orderbook's `userOrders` map.

Only resting open or partially filled orders are in `userOrders`.

## Snapshots

The engine snapshots successful mutating operations.

Default path:

```txt
snapshots/core-engine.snapshot.txt
```

Override path:

```sh
ENGINE_SNAPSHOT_PATH=/some/path/snapshot.json bun run start
```

Snapshot data includes:

- `eventSequenceId`
- `balances`
- `markets`
- `positions`
- `orders`

`bigint` values are stored as strings because JSON cannot encode BigInts.

On startup:

1. `Engine` checks whether the snapshot file exists.
2. If it exists, `loadSnapshot()` parses it.
3. Balances are rebuilt with BigInts.
4. Markets are rebuilt.
5. Positions are rebuilt with BigInts.
6. Orders are rebuilt with BigInts.
7. Empty orderbooks are created for every market.
8. Open or partially filled orders with remaining quantity are restored into the right orderbook.
9. If no snapshot exists, default markets are initialized.

This means the live book is reconstructed from persisted order history, while filled and cancelled orders remain queryable through the global order store.

## Response Normalization

The engine must never return raw BigInts to JSON callers. `normalizeOrderReturn(...)` converts:

- order quantity
- entry price
- filled quantity
- remaining quantity
- average price
- perp margin
- fill price
- fill quantity
- fill trade id
- depth price
- depth quantity

All these values are returned as strings.

## Error Handling

Business rejections throw `RejectError` with an `EVENT_REJECT_CODES` value. The core engine catches these and returns:

```ts
{
  success: false,
  eventId,
  timestamp,
  message,
  code
}
```

Unexpected errors are logged and returned as internal engine failures.

Common reject codes include:

- `INVALID_MARKET`
- `INVALID_SIDE`
- `INVALID_ORDER_TYPE`
- `INVALID_QUANTITY`
- `INVALID_PRICE`
- `INSUFFICIENT_BALANCE`
- `INSUFFICIENT_MARGIN`
- `ORDER_NOT_FOUND`
- `POST_ONLY_WOULD_TRADE`
- `STP_TRIGGERED`
- `NO_LIQUIDITY`
- `MARKET_NOT_EMPTY`

## Important Invariants

- All internal money and quantity calculations use `bigint`.
- Incoming JSON amounts are strings.
- Outgoing JSON amounts are strings.
- Funds are locked before matching.
- Spot fills debit locked balances.
- Perp fills update positions and keep margin locked while exposure remains.
- Resting orders exist in both market-local indexes and global lookup maps.
- Filled makers are removed from live orderbook indexes but kept in global order history.
- Mutating successful events save snapshots.
- Query-only events do not save snapshots.

## Current Implementation Notes

The engine is intentionally in-memory with snapshot persistence. It is suitable for local development and deterministic engine iteration. For production-grade exchange operation, the next pieces to add would usually be:

- durable append-only event log
- idempotency keys / client order ids
- stronger snapshot atomicity
- fees
- funding payments for perps
- mark/index price feeds
- liquidation engine
- richer margin modes
- admin authorization around market mutations
- formal test coverage for matching and settlement edge cases
